# 中文程式碼審查（zh-code-review）

## 觸發時機

- 用戶說：「幫我 review 這段程式碼」「看一下這個 PR」「code review 一下」
- 用戶提供檔案路徑、diff 或選取程式碼片段

## 審查面向

依序檢視（有問題才寫，無則可簡述「未發現明顯問題」）：

1. **正確性**：邏輯錯誤、邊界條件、錯誤處理、併發與狀態一致性
2. **可讀性**：命名、結構、重複、模組邊界、註解是否必要且清晰
3. **效能**：明顯 N+1、不必要配置/分配、熱路徑阻塞
4. **安全性**：注入、鑑權、敏感資料外洩、依賴風險、輸入驗證

## 輸出格式

使用以下三類標題分類（繁簡與用戶語系一致）：

```markdown
## 總評

<一兩句話：是否建議合併，以及前提>

## ✅ 優點

- ...

## ⚠️ 建議改善

- **[檔案:行號]** 說明與具體建議

## ❌ 需要修正

- **[檔案:行號]** 阻塞合併的問題與修正方向
```

## 輸出範例

```markdown
## 總評

邏輯大致正確，修正錯誤處理與一處潛在 race 後可合併。

## ✅ 優點

- 將付款回調抽成獨立 handler，職責清楚
- 單元測試涵蓋成功與簽章失敗情境

## ⚠️ 建議改善

- **[src/order.ts:42]** 魔術數字 `3000` 建議抽成具名常數 `PAYMENT_TIMEOUT_MS`
- **[src/order.ts:88]** 日誌可加上 `orderId` 方便追查

## ❌ 需要修正

- **[src/webhook.ts:31]** 未驗證 webhook 簽章，存在偽造請求風險；應使用 Stripe SDK 驗證
- **[src/order.ts:105]** `markPaid` 非冪等，重複事件可能重複更新；需依 `event.id` 去重
```

## 原則

- 每條意見盡量指向**具體位置**與**可執行建議**。
- 區分阻塞（❌）與非阻塞（⚠️），避免堆砌風格偏好。
- 技術術語保留英文（如 `branch`、`PR`、`middleware`）。
- 用戶未要求時，不直接改動程式碼，除非用戶希望一併修復。
